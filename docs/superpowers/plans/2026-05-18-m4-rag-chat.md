# M4 RAG-Chat Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Granularity note:** Unlike typical writing-plans output, this plan operates at **agent-dispatch + worktree-PR granularity** rather than per-TDD-step. The playground project's discipline is "every code change = one worktree → one subagent dispatch → one PR". Each task's steps follow that loop. Inside each subagent dispatch, the subagent does its own TDD decomposition per its agent definition.

**Goal:** Ship M4 (auth-only Perplexity-style chat over the playground corpus) end-to-end — PRD, ADR-14 + 6 doc amendments, design, GitHub issues, and three implementation slices (infra → backend → frontend) for each functional layer (s1 single-shot, s2 sessions, s3 polish).

**Architecture:** New `rag-chat` BC with the standard four-module quadruplet (`rag-chat-{api,app,domain,infra}`). SSE streaming endpoint `POST /api/rag/chat` backed by Spring AI 1.0 `ChatClient` → Qwen3-32B via `spark-inference-gateway`. Per-turn retrieval embeds the current user message via BGE-M3 and runs a parameterized vector SQL against M3's `rag.document_chunks` (cross-schema SELECT). New `chat` schema for `sessions`, `messages`, `message_citations`. Frontend ships a full-page `/chat` route with top-tab session navigation and inline citation accordions; no side panel. Auth-only — anon redirects to `/login?return=/chat`.

**Tech Stack:**
- **Backend:** Spring Boot 3.3.x, Spring AI 1.0 GA (`spring-ai-openai-spring-boot-starter` for the gateway proxy), Spring Modulith (for the BC scaffold, no event publishing), Postgres 16 + pgvector (read M3's chunks, write `chat` schema), Redis + Redisson (rate-limit token bucket), Resilience4j (circuit breaker), Flyway (migrations).
- **Frontend:** Next.js 14 App Router + React Server Components, `EventSource` (browser-native SSE consumption — no library), `unified` + `remark-gfm` + `rehype-sanitize` + `shiki` (Markdown rendering reused from M2), `remark` plugin for `[N]` marker → citation pill replacement (new).
- **Infra:** `infra/docker-compose.yml` adds `rag-chat-playground` service (port 18084 candidate, ADR-14 pins); Flyway init for `chat` schema; Redis namespace `rag-chat:`.

## Risk / dependency callouts

- **M3 dependency:** Tasks 5+ block on M3 being mergeable (pgvector populated with chunks). If M3 stalls, M4 implementation also stalls but PRD/ADR/Design (Tasks 1–4) can still progress.
- **M2 ongoing in parallel:** the M2 implementation session is shipping slices (m2-s1 done; m2-s2 may be in flight). M4 frontend reuses M2's Markdown render pipeline — if M2 changes the pipeline contract, Task 7's `citationMarker.ts` plugin may need adjustment. Flag if M2 publishes breaking changes to the Markdown renderer.
- **Spring AI 1.0 GA contract:** stable as of ADR-04 confirmation, but Spring AI minor versions can change `ChatClient` / `EmbeddingModel` API shapes. ADR-14 pins the version range.
- **TalkToFigma channel lifetime:** Task 4 (design) requires the bun socket bridge to be alive and the Figma plugin to be joined to a fresh channel each session. See `docs/infra-requirements/talk-to-figma.md` for the Spark-local Firefox setup.
- **Cost ceiling:** M4 is the first milestone that exercises Qwen3-32B in production-style streaming load. The auth-only 60 completions/hour cap (ADR-14) is the cost backstop — Task 10 wires this. If real usage approaches this regularly, ADR-14 should be amended to revise.
